A More Realistic Example

Let's look at a more realistic instance: a customer inputs their customer name and password, the algorithm we're evaluating checks if both are correct by checking the matched inputs against stored pairs of customer names and passwords. If both are correct, the customer can view the progress of their order, if not, the customer cannot view their order. If one of the pair matches any one of either the customer names or the passwords, the customer is offered the opportunity to answer a security question.

How many tests cases do we need to plan to provide a reasonable evaluation of the design?

2 for correct and incorrect customer names

2 for correct and incorrect passwords

1 for correct customer name and incorrect password pairs

1 for incorrect customer name and correct password pairs

Now we'll include a restriction on the length of passwords? Say a minimum of six characters can be used. Our algorithm checks for minimum and below minimum length passwords. How many tests do we need for this?

1 for less than six characters

1 for exactly six characters

How many tests will we need to plan for combinations of these conditions? Ideally we should test for each condition on the length of the password for each of the possible combinations above. That is, for example, for the test of a correct customer name we should plan to test.

Correct customer name - test with an incorrect password of length minimum -1

Correct customer name - test with an incorrect password of length minimum

We would repeat this for incorrect customer names and so on. As you can see the number of planned tests increases very quickly.
