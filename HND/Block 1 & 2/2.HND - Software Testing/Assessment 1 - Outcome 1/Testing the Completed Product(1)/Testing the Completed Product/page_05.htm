Testing Strategies

Before deciding on which approaches to testing we'll take, whether to treat the software, or artefact, as a black box, or with its inner workings completely visible, we need to decide on a testing strategy.

We assume that strategy means "the order in which things are performed".

Look here for an idea of the many differing definitions of the term strategy:

http://www.google.co.uk/search?hl=en&lr=lang_en&as_qdr=all&defl=en&q=define:strategy&sa=X&oi=glossary_definition&ct=title

Next: Some important testing strategies

We will look at three possible strategies:

  • Top down testing
  • Bottom up testing
  • Thread testing

In Top down testing we start at the outermost (or top) layer, test this, then move down, testing the next outermost (or next down from the top) until everything has been tested. For example, if using a top down strategy for testing the degree to which a shelter and safe, containing a box, inside the shelter, were all bomb-proof, the shelter would be tested first followed by the safe and lastly the box.

In Bottom up testing we start at the innermost (or bottom) layer and test, then move up (or out) a layer and test again. In this case, we would test whether the box was bomb-proof, then the safe and then the shelter.

In Thread testing we follow a typical interaction, testing as we follow the 'thread' of the interaction. Thread testing requires a domain specific scenario, for example, a scientist places the secret formulae in the bomb-proof safe (ignoring the box as unnecessary), ensures the bomb-proof safe is in the bomb-proof shelter and [...goes on her way...]. We would test the safe, followed by the shelter. You will have noticed that by the end of this set of tests, we have not evaluated (tested) if the box is bomb-proof because this is not part of the specified interaction.
