Testing Strategy

The recommended strategy for testing an application is top-down; here the high levels of the system would be tested before testing the detailed components. In this strategy, each individual graphic interface is tested as soon as it is created, despite having a very limited functionality. In other words, the various components of the individual interface such as buttons, links, text fields, etc are tested first. In this way the application is represented as a single abstract component with further sub-components that are hierarchically built and tested one after the other. The same strategy is applied to all modules as they are created: that is, after the top-level component has been tested, its sub-components are implemented and tested in the same way. This process continues recursively until the bottom-level components are implemented. The whole system may then be completely tested.

Top-down testing is used with top-down programming development so that a system component is tested as soon as it is coded. Coding and testing almost become a single activity, with no separate component or module-testing phase. In this way, unnoticed design errors may be detected at an early stage in the testing process. As these errors are usually structural errors, early detection means that extensive redesign and/or re-implementation may be avoided. It also has the further advantage of having a limited working system available at a very early stage. Validation can begin early in the testing process, as a demonstrable system can be made available to the users.

Splitting the whole system up in a modular way helps us develop programs or systems more easily. Module testing is a series of tests applied to a particular module to ensure that it works before being joined on to any other module. Modular testing is used because it's easier to test individual modules before they are allowed to interact with other modules. If more than one module is being debugged, this makes the debugging process much more complex.
